# tuimux PRD (Product Requirements Document)

- **문서 버전**: 0.1 (Draft)
- **작성일**: 2026-06-08
- **상태**: 검토용 초안 (구현 코드 없음, 제품/아키텍처 의사결정 문서)
- **프로젝트명**: tuimux
- **한 줄 요약**: prefix 키 없이, 마우스와 우측 탭으로 직관적으로 다룰 수 있는 Rust 기반의 "의도적으로 단순한" tmux 대체 세션 매니저.

---

## 1. 개요 (Overview)

tuimux는 터미널 멀티플렉서이자 세션/프로세스 매니저다. tmux의 핵심 가치(여러 셸/프로세스를 한 화면에서 관리, 분리(detach)/재접속(attach), 백그라운드 유지)는 계승하되, tmux의 진입 장벽(prefix 키 조합, 난해한 설정, 명령어 암기)을 제거하는 것을 목표로 한다.

핵심 차별점:

1. **No prefix-key**: `Ctrl-b`, `Ctrl-a` 같은 prefix 개념이 없다. 모든 조작은 마우스 또는 직관적인 단일 키/메뉴로 수행한다.
2. **Full TUI**: 기능이 명령행 옵션이나 숨겨진 단축키가 아니라, 화면에 보이는 TUI 위젯(탭, 패널, 버튼, 리스트)으로 노출된다.
3. **실행 중에도 보이는 프로세스 관리**: 명령/프로세스가 돌고 있는 동안에도 TUI 상에서 해당 프로세스 목록·상태·자원을 보고 제어할 수 있다.
4. **우측 세로 탭**: 세션/창 전환을 화면 우측의 세로 탭 바로 처리해, 클릭 한 번으로 전환한다.
5. **Full mouse support**: 탭 클릭, 패널 포커스, 분할 경계 드래그, 스크롤, 컨텍스트 메뉴까지 마우스로 가능.

이 문서는 PRD다. 구현 코드는 포함하지 않으며, 무엇을/왜 만드는지와 아키텍처 선택지의 트레이드오프를 정리한다.

## 2. 목표 (Goals)

- **G1. 학습 비용 최소화**: tmux를 모르는 사용자가 매뉴얼 없이 5분 안에 세션 생성·분할·전환·detach/attach를 할 수 있다.
- **G2. 발견 가능성(Discoverability)**: 가능한 모든 동작이 화면에 보이거나, 보이는 진입점(우클릭 메뉴, 버튼, 도움말 오버레이)에서 도달 가능하다.
- **G3. 실행 중 가시성**: 장시간 실행되는 명령(빌드, 학습, 서버)이 돌아가는 중에도 프로세스 상태(PID, 실행시간, exit 여부)를 TUI에서 확인·종료할 수 있다.
- **G4. 세션 지속성**: 사용자가 SSH 연결을 끊거나 클라이언트를 종료해도 세션과 그 안의 프로세스는 계속 살아 있고, 재접속 시 그대로 복구된다.
- **G5. 마우스 우선, 키보드 동등**: 모든 마우스 동작에는 동등한 키보드 단축키가 있어 마우스 없는 환경에서도 100% 사용 가능하다.
- **G6. 단일 바이너리 배포**: 의존성 최소화, 정적/단일 실행 파일 지향으로 `scp` 한 번으로 원격 서버에 설치 가능.

### 성공 지표 (Success Metrics)

- 신규 사용자가 문서 없이 핵심 작업 6종(세션 생성/분할/전환/detach/attach/프로세스 종료)을 성공한 비율 ≥ 90%.
- detach 후 SSH 재접속 시 세션 복구 성공률 100% (지원 환경 한정).
- 첫 화면 렌더링까지 < 100ms, 입력 지연 체감 없음(키 입력→화면 반영 < 16ms 목표).

## 3. 비목표 (Non-Goals)

- **NG1. tmux 설정/스크립트 호환성**: `.tmux.conf`, tmux 명령어 문법, tmux 플러그인(tpm 등) 호환은 목표가 아니다.
- **NG2. 무한 커스터마이즈**: tmux/screen 수준의 끝없는 옵션 트리를 제공하지 않는다. "합리적 기본값 + 소수의 선택지"를 지향한다.
- **NG3. 원격 멀티플렉싱 프로토콜 자체 개발(초기)**: 자체 네트워크 프로토콜로 원격 attach를 구현하지 않는다. 원격 사용은 "SSH 접속 후 로컬 tuimux 실행" 모델을 따른다.
- **NG4. Windows 네이티브 지원(초기)**: 초기 타깃은 Linux. macOS는 best-effort. Windows(ConPTY)는 후순위.
- **NG5. 터미널 에뮬레이터 교체**: tuimux는 터미널 에뮬레이터(alacritty/kitty 등) 안에서 동작하는 멀티플렉서이지, GPU 렌더링 터미널 에뮬레이터 자체가 아니다.
- **NG6. 세션 내용의 영구 디스크 기록(초기)**: 스크롤백을 디스크에 영구 저장/세션 복원(reboot 후 프로세스 부활)은 초기 범위 밖. 지속성은 "프로세스가 살아있는 동안"에 한정.

## 4. 페르소나 (Personas)

### P1. "원격 서버 개발자" — 지수
- 노트북에서 SSH로 클라우드 VM에 접속해 작업한다.
- 빌드/학습을 돌려놓고 노트북을 닫거나 카페로 이동(네트워크 끊김)한다.
- tmux를 쓰긴 하는데 `Ctrl-b` 다음에 뭘 눌러야 하는지 매번 헷갈린다.
- **니즈**: 끊겨도 죽지 않는 세션 + 외울 것 없는 조작.

### P2. "터미널 입문 주니어" — 민호
- 부트캠프 출신, GUI에 익숙하고 터미널 단축키 암기를 싫어한다.
- 마우스로 탭을 클릭하고 패널 경계를 드래그하길 원한다.
- **니즈**: GUI처럼 보이고 만져지는 멀티플렉서.

### P3. "운영/SRE" — 서연
- 여러 서버에 들어가 동시에 로그를 보고 프로세스를 죽이고 재시작한다.
- 어떤 패널에서 뭐가 돌고 있는지(PID, 점유 자원) 한눈에 보고 싶다.
- **니즈**: 실행 중 프로세스의 가시성과 즉각적 제어.

### P4. "도구 까다로운 시니어" — (반대 페르소나)
- 이미 tmux를 손에 익혔고 prefix가 불편하지 않다.
- tuimux를 쓸 이유는 "새 환경에서의 빠른 셋업"과 "팀 주니어 온보딩" 정도.
- **함의**: 파워유저를 뺏는 게 목표가 아니라, 진입 장벽 때문에 멀티플렉서를 안 쓰던 층을 끌어들이는 게 목표.

## 5. 요구사항 (Requirements)

표기: **[M]** = MVP 필수, **[L]** = 이후(Later), **[O]** = 선택(Optional).

### 5.1 세션/창/패널 모델
- **R-S1 [M]** 서버(데몬) 하나가 여러 **세션(session)** 을 보유한다.
- **R-S2 [M]** 세션은 여러 **창(window)** 을, 창은 여러 **패널(pane)** 을 가진다. (tmux와 동일한 3계층 모델)
- **R-S3 [M]** 각 패널은 하나의 PTY와 그 위에서 도는 프로세스 트리를 가진다.
- **R-S4 [M]** 패널 수평/수직 분할, 닫기, 크기 조절.
- **R-S5 [L]** 패널을 다른 창/세션으로 이동(이동/스왑).
- **R-S6 [O]** 패널 줌(임시 전체화면), 동기 입력(여러 패널 동시 타이핑).

### 5.2 지속성 / detach·attach
- **R-P1 [M]** 클라이언트 종료/연결 끊김과 무관하게 서버와 세션이 유지된다.
- **R-P2 [M]** `tuimux attach`(또는 인자 없이 실행 시 자동 attach)로 기존 세션에 재접속.
- **R-P3 [M]** detach 시 패널 내 프로세스는 SIGHUP을 받지 않고 계속 실행된다(PTY 소유권은 서버가 유지).
- **R-P4 [M]** 여러 클라이언트가 같은 세션에 동시 attach 가능(미러링). 리사이즈 정책 정의 필요(§9).
- **R-P5 [L]** OS 재부팅 후 자동 세션 부활(프로세스 재실행 스크립트 기반). 초기엔 비목표(NG6).

### 5.3 프로세스 가시성/관리 (차별화 핵심)
- **R-X1 [M]** 사이드 패널 또는 오버레이에서 현재 세션의 모든 패널과 그 안에서 실행 중인 **포그라운드 프로세스**(명령줄, PID, 실행시간)를 리스트로 표시.
- **R-X2 [M]** 프로세스가 종료되면 패널 탭/리스트에 종료 상태(exit code, 성공/실패 색상)를 표시.
- **R-X3 [M]** 리스트에서 프로세스를 선택해 신호 전송(SIGINT/SIGTERM/SIGKILL)을 메뉴로 보냄.
- **R-X4 [L]** 패널별 CPU/메모리 점유 표시(주기적 `/proc` 폴링 기반).
- **R-X5 [L]** "실행 완료/실패 시 알림"(탭 깜빡임, 벨, 데스크톱 알림 연동).

### 5.4 UI / 입력
- **R-U1 [M]** 화면 **우측 세로 탭 바**: 창(또는 세션) 목록을 세로로 나열, 클릭/단축키로 전환.
- **R-U2 [M]** **No prefix**: 전역 동작은 단일 modifier 키(예: `Alt-…`) 또는 마우스로 수행. prefix 시퀀스 없음.
- **R-U3 [M]** **Full mouse**: 탭 클릭 전환, 패널 클릭 포커스, 분할 경계 드래그 리사이즈, 휠 스크롤(스크롤백), 우클릭 컨텍스트 메뉴.
- **R-U4 [M]** 항상 보이는 **상태 바**(세션명, 시간, 접속 클라이언트 수)와 호출 가능한 **도움말 오버레이**(`?`).
- **R-U5 [M]** 명령 팔레트(Command Palette) 스타일의 단일 진입점(`Alt-p` 등)에서 모든 동작 검색·실행.
- **R-U6 [O]** 텍스트 선택/복사(마우스 드래그 → 클립보드, OSC 52 연동).
- **R-U7 [O]** 색 테마(라이트/다크), 폰트 무관 ASCII/Unicode 경계 문자 토글.

### 5.5 비기능 요구사항 (NFR)
- **NFR-1** 입력 지연: 키→렌더 < 16ms(60fps 체감) 목표.
- **NFR-2** 자원: 유휴 시 CPU ≈ 0%(이벤트 기반, busy-loop 금지), 데몬 메모리 풋프린트 작게.
- **NFR-3** 안정성: 클라이언트 패닉이 서버/세션을 죽이지 않는다(프로세스 분리).
- **NFR-4** 호환성: `$TERM`에 따른 기능 차등, truecolor/256색/모노 폴백.
- **NFR-5** 보안: 서버 IPC 소켓은 사용자 전용 권한(`0700` 디렉터리, `0600` 소켓), 다른 사용자가 attach 불가.

## 6. UX 설계 (User Experience)

### 6.1 화면 레이아웃
```
┌──────────────────────────── status bar ───────────────────────────┐
│ tuimux · session: dev · 2 clients · 14:03                          │
├───────────────────────────────────────────────────────┬───────────┤
│                                                         │  TABS     │
│   pane 0 (focused)                  │  pane 1           │ ▸ 1 build │
│   $ cargo build                     │  $ htop          │   2 logs  │
│   Compiling tuimux ...              │                   │   3 ssh   │
│                                     │                   │  ───────  │
│─────────────────(drag to resize)────│                   │  + new    │
│   pane 2                            │                   │           │
│   $ tail -f app.log                 │                   │  PROCS    │
│                                     │                   │ ● build…  │
│                                     │                   │ ● htop    │
│                                     │                   │ ✓ test ✓  │
├─────────────────────────────────────────────────────────┴──────────┤
│  [?] help   [Alt-p] palette   [Alt-n] new   [Alt-w] close          │
└────────────────────────────────────────────────────────────────────┘
```
- **우측 상단 TABS**: 창/세션 세로 탭. 현재 탭은 `▸`로 표시, 클릭 전환, `+ new`로 생성.
- **우측 하단 PROCS**: 실행 중/완료 프로세스 패널(R-X1~3). `●` 실행중, `✓` 성공, `✗` 실패.
- **하단 힌트 바**: 자주 쓰는 동작을 항상 노출(발견 가능성).

### 6.2 핵심 인터랙션
- **세션 전환**: 우측 탭 클릭 또는 `Alt-1..9` / `Alt-↑↓`.
- **분할**: 우클릭 메뉴 "Split right / Split down" 또는 `Alt-|`, `Alt--`.
- **리사이즈**: 패널 경계에 마우스 올리면 커서 변화 → 드래그. 키보드는 `Alt-Shift-방향키`.
- **detach**: 상태 바의 detach 버튼 클릭 또는 `Alt-d`. detach 후 셸로 빠져나오고 세션은 백그라운드 유지.
- **프로세스 종료**: PROCS 리스트에서 항목 우클릭 → "Send SIGINT/TERM/KILL".
- **도움말**: `?` 누르면 전체 단축키 오버레이(마우스로도 닫기).

### 6.3 No-prefix 키 정책
- 전역 동작은 `Alt`(Meta) 조합으로 통일 → 패널 내부 셸/앱의 `Ctrl` 키와 충돌 최소화.
- `Alt` 키가 막힌 터미널을 위해 **명령 팔레트**(`Alt-p`, 대체로 `Ctrl-Space` 폴백)와 **마우스 메뉴**라는 두 개의 prefix-free 경로를 항상 제공.
- 충돌이 불가피한 일부 환경을 위해 단축키 리매핑 설정(소수 항목)만 허용(NG2 범위 내).

## 7. MVP 범위 (MVP Scope)

**MVP는 "혼자 쓰는 1대 서버에서, 끊겨도 살아있고, 마우스로 다룰 수 있는 멀티플렉서"** 다.

포함:
- 단일 사용자 로컬 데몬 + 클라이언트 분리 구조 (§8 옵션 1).
- 세션/창/패널 3계층, 분할·전환·리사이즈·닫기.
- detach/attach, SSH 끊김 생존(§9 핵심).
- 우측 세로 탭, 전체 마우스 지원, 상태 바, 도움말 오버레이, 명령 팔레트.
- PROCS 패널(실행 중 표시 + 종료 상태 + 신호 전송).
- 스크롤백(메모리 내, 고정 라인 수) + 휠 스크롤.

제외(이후):
- 다중 클라이언트 동시 attach의 정교한 리사이즈 협상(초기엔 단순 정책).
- 자원 사용량(CPU/메모리) 표시, 알림.
- 패널 이동/스왑, 줌, 동기 입력.
- 세션 레이아웃/스크롤백 디스크 영속화, 재부팅 후 부활.
- macOS 정식 지원, Windows.

**MVP 완료 정의(DoD)**: SSH로 접속 → tuimux 실행 → 빌드 명령 시작 → SSH 강제 종료 → 재접속 → `tuimux attach` → 빌드가 계속 진행 중이고 출력이 이어짐. 이 시나리오가 100% 재현되면 핵심 검증 통과.

## 8. 아키텍처 옵션 (Architecture Options)

tuimux의 가장 큰 기술 결정은 **"터미널 멀티플렉싱 엔진을 직접 만들 것인가, tmux를 백엔드로 쓸 것인가"** 다. 두 옵션을 비교한다.

### 옵션 1 — 네이티브 Rust 세션/서버 + 자체 PTY 관리

구조: `tuimux`는 (a) 백그라운드 **서버/데몬** 프로세스와 (b) 사용자가 보는 **클라이언트(TUI)** 로 나뉜다. 둘은 사용자 전용 **유닉스 도메인 소켓**으로 통신한다.

- **서버 책임**: 각 패널마다 PTY 마스터를 `openpty`로 열고, 슬레이브에 셸/명령을 `fork+exec`. PTY 마스터의 출력을 읽어 **터미널 에뮬레이션(VT100/xterm 파서)** 으로 화면 셀 그리드(스크린 버퍼)에 반영. 클라이언트가 attach하면 현재 그리드 상태와 이후 변경분을 전송. 클라이언트 입력은 해당 PTY로 write.
- **클라이언트 책임**: 서버에서 받은 셀 그리드를 렌더(우측 탭/마우스/상태바 등 tuimux UI 포함), 키/마우스 이벤트를 서버로 전달. 클라이언트가 죽어도 서버는 무관.
- **핵심 구성요소**: PTY 관리, VT 파서(스크린 버퍼/스크롤백/속성), 레이아웃 엔진, IPC 프로토콜, TUI 렌더러.
- **Rust 생태계(후보, 미확정)**: PTY는 `nix`/`rustix`의 `openpty`·`forkpty` 또는 `portable-pty`. VT 파싱은 `vte`(alacritty의 파서) 또는 `wezterm-term`/`vt100` 같은 셀-그리드 라이브러리. TUI 렌더는 `ratatui` + `crossterm`. 비동기 I/O는 `tokio` 또는 `mio` 기반 이벤트 루프. (선택은 설계 단계에서 벤치 후 확정.)

**장점**
- UX 완전 제어: prefix-free·우측 탭·실행 중 프로세스 패널 같은 차별화 기능을 자연스럽게 구현.
- 단일 바이너리, 외부 런타임 의존 없음(tmux 미설치 서버에서도 동작).
- 데몬 동작·소켓 권한·detach 의미를 우리가 정의 → 지속성(§9) 정책을 명확히 통제.

**단점**
- **VT 에뮬레이션이 가장 큰 비용**: xterm 호환은 끝이 없는 디테일(스크롤 영역, 대체 화면, 마우스 리포팅 패스스루, 와이드/조합 문자, 색/속성, 비프, 타이틀 OSC 등). 여기서 호환성 버그가 사용성 체감을 좌우.
- 멀티플렉서의 어려운 부분(리사이즈/리플로우, 여러 클라이언트 동기화)을 직접 떠안음.

### 옵션 2 — tmux를 백엔드로 감싸는 풀 TUI 래퍼

구조: 실제 멀티플렉싱/PTY/지속성은 **tmux 서버**가 담당하고, tuimux는 그 위에 얹는 **UI/제어 레이어**.

세부 방식 두 가지:
- **2a. control mode 래퍼**: `tmux -CC`(control mode)로 tmux 서버에 붙어, tmux가 내보내는 출력/이벤트를 파싱해 tuimux UI로 렌더하고, 사용자 동작을 tmux 명령으로 변환해 보낸다. (iTerm2의 tmux 통합과 동일 계열 접근.)
- **2b. 명령 오케스트레이션**: `tmux new-session/split-window/send-keys/list-panes …`를 호출해 세션을 구성·조회하고, 화면 자체는 tmux client를 임베드하거나 control mode로 받는다.

**장점**
- 성숙한 VT 에뮬레이션·지속성·리사이즈를 **공짜로** 얻음. 호환성 리스크가 가장 큰 부분을 외주화.
- 지속성(§9)은 tmux 서버가 이미 잘 해결한 문제 → "끊겨도 생존"이 기본 제공.
- 초기 개발 속도가 빠르다(엔진을 안 만듦).

**단점**
- **차별화 기능이 tmux 모델에 갇힘**: prefix-free·우측 탭·실행 중 프로세스 패널은 tmux의 추상(창/패널) 위에서 재현은 가능하나, control mode가 노출하는 정보·이벤트 한계에 종속.
- **외부 의존**: 대상 서버에 호환되는 tmux 버전이 깔려 있어야 함(NG의 단일 바이너리 목표와 충돌). 버전별 control mode 차이 대응 비용.
- **프로세스 가시성(R-X)** 은 tmux가 직접 안 줌 → 어차피 `pane_pid`로부터 `/proc` 트리를 우리가 따로 조회해야 함(즉 옵션 1과 동일 작업이 일부 필요).
- 두 레이어(tuimux↔tmux) 사이의 임피던스 불일치로 미묘한 버그·지연 발생 가능.

### 권고 (Recommendation)
- **MVP는 평가 스파이크(spike)를 먼저** 둔다: 옵션 2a(tmux control mode)로 "우측 탭 + 마우스 + detach 생존"을 빠르게 프로토타이핑해 UX 가설을 검증한다.
- **장기적으로는 옵션 1(네이티브)** 로 간다. 차별화의 본질(완전한 UX 제어, 단일 바이너리, 실행 중 프로세스 패널, prefix-free)은 자체 엔진에서만 온전히 달성된다. VT 에뮬레이션은 성숙한 Rust 크레이트(`vte`/`vt100`/`wezterm-term`)를 채택해 직접 작성 위험을 낮춘다.
- 즉, **"control mode로 UX를 증명 → 네이티브로 본 구현"** 의 2단계 전략. 두 경로 모두에서 PROCS 패널은 `/proc` 기반으로 우리가 만든다.

## 9. 핵심 기술 질문: "SSH 끊겨도 사용자 세션이 살아있는가?" (Persistence)

**짧은 답: 그렇다 — 단, 제대로 설정했을 때.** 사용자 권한(root 아님)만으로도 SSH 연결이 끊긴 뒤에도 tuimux 세션과 그 안의 프로세스를 살릴 수 있다. 핵심은 **세션을 SSH 로그인 세션의 수명에서 분리(decouple)하는 것**이다. tmux/screen이 수십 년간 해 온 일과 본질적으로 같다.

### 9.1 왜 보통은 죽는가
SSH로 붙으면 로그인 셸과 그 자식들은 하나의 **세션/프로세스 그룹**, 그리고 SSH가 할당한 **PTY**에 묶인다. 연결이 끊기면:
1. SSH 데몬이 PTY 마스터를 닫는다.
2. 커널이 제어 터미널의 세션 리더에게 **SIGHUP**을 보내고, 전경 프로세스 그룹으로 전파된다.
3. SIGHUP을 받은 프로세스들이 종료된다.

즉 죽는 이유는 (a) **제어 터미널 소멸 + SIGHUP**, (b) **부모(셸)의 종료에 묶인 수명** 때문이다. 두 고리를 끊으면 생존한다.

### 9.2 tuimux의 생존 메커니즘 (PTY 소유권 + 데몬)
- **별도 데몬이 PTY를 소유**: tuimux 서버(데몬)가 각 패널의 **PTY 마스터를 직접 보유**한다. 패널 안 프로세스의 제어 터미널은 SSH의 PTY가 아니라 **tuimux가 연 PTY**다. 따라서 SSH 연결이 끊겨 SSH PTY가 닫혀도, 패널 프로세스의 제어 터미널은 멀쩡하다 → SIGHUP 안 옴.
- **데몬을 로그인 세션에서 분리**: 서버는 처음 뜰 때 `setsid()`로 **새 세션 리더**가 되어 SSH의 제어 터미널과 세션에서 떨어져 나온다(자기 자신은 제어 터미널 없음). 그러면 SSH 끊김 시의 SIGHUP 전파 사슬 밖에 있게 된다.
- **클라이언트만 SSH PTY를 쓴다**: 사용자가 보는 클라이언트(TUI)만 SSH PTY에 그려진다. 연결이 끊기면 클라이언트는 죽지만(또는 명시적 detach), 서버·PTY·프로세스는 산다. 재접속해 `attach`하면 클라이언트가 새로 붙는다.
- 이는 옵션 1·2 모두에 적용된다. 옵션 2(tmux 백엔드)에서는 tmux 서버가 정확히 이 역할을 한다.

### 9.3 데몬을 어떻게 "계속" 살리나 — 세 가지 수준
끊김 순간의 SIGHUP은 9.2로 막힌다. 그런데 "사용자가 **완전히 로그아웃**(모든 세션 종료)" 하거나 시스템이 사용자 프로세스를 정리하려 할 때까지 살리려면 추가 고려가 필요하다.

1. **데몬 자가 분리(가장 단순, MVP)**
   - 서버를 `fork` → 부모 종료 → 자식이 `setsid()`로 데몬화, 표준 입출력 정리, 사용자 런타임 디렉터리(`$XDG_RUNTIME_DIR` 또는 `~/.tuimux`)에 소켓 생성.
   - tmux/screen과 동일한 고전적 방식. **추가 설정 없이** SSH 끊김·detach 후 생존.
   - 한계: 아래 9.4의 **사용자 인스턴스 정리(systemd KillUserProcesses / lingering)** 정책에 종속.

2. **systemd `--user` 서비스로 등록(권장, 견고)**
   - tuimux 데몬을 **systemd user service**(`systemctl --user start tuimux`)로 띄우면, 데몬이 사용자 로그인 셸이 아니라 **user systemd 인스턴스**의 자식이 되어 로그인 세션과 더 깔끔히 분리된다.
   - 재시작 정책(`Restart=on-failure`), 로깅(journal), 소켓 활성화 등 운영 이점.

3. **linger 활성화(완전 무세션 생존)**
   - `loginctl enable-linger <user>` 를 켜면, 해당 사용자의 **마지막 로그인 세션이 끝나도** user systemd 인스턴스가 계속 떠 있다 → user service인 tuimux 데몬도 계속 생존.
   - linger를 안 켜면, 배포판 설정에 따라 **마지막 세션 종료 시 user 인스턴스와 그 자식들이 정리**될 수 있다(아래 9.4).

### 9.4 함정: `systemd-logind`의 `KillUserProcesses` / lingering
사용자 수준 지속성의 진짜 변수는 **로그아웃 시 남은 프로세스를 죽이느냐**다.

- `logind.conf`의 **`KillUserProcesses`**:
  - `yes`(일부 배포판 기본): 사용자의 **마지막 세션이 끝나면** 그 사용자의 잔여 프로세스를 **모두 죽인다**. 이 경우 9.3-(1)의 단순 데몬화만으로는 "완전 로그아웃 후"에 살아남지 못할 수 있다. → **linger가 필요**하거나, 프로세스를 user-slice가 아닌 곳으로 빼야 함.
  - `no`(다수 배포판 기본): 세션이 끝나도 잔여 프로세스를 죽이지 않음 → 단순 데몬화로도 생존.
  - 예외 화이트리스트(`KillExcludeUsers`, 보통 root) 존재.
- 정리하면 **두 축**이 결정한다:
  1. SSH **끊김(연결만 끊고 로그인 세션은 살아있을 수 있음)** vs **완전 로그아웃**.
  2. `KillUserProcesses` 값 + **linger** 여부.
- **권장 조합**: tuimux를 **user systemd service**로 돌리고, "로그아웃해도 살리겠다"는 사용자에겐 **`loginctl enable-linger`** 를 안내. 이 조합이면 `KillUserProcesses` 값과 무관하게 사용자 권한만으로 견고하게 생존.

> 주의: SSH "연결 끊김"과 "완전 로그아웃"은 다르다. 단순 연결 끊김(다른 세션이 남아있거나, 단지 detach)에서는 대개 9.2/9.3-(1)만으로 충분하다. "모든 세션을 닫은 뒤에도" 살리는 게 목표라면 linger를 권장한다.

### 9.5 PTY 소유권/한계 (Limits)
- **PTY 개수 상한**: 패널마다 PTY 1개. 시스템 PTY 수는 `/proc/sys/kernel/pty/max`(전역)와 파일 디스크립터 한계(`ulimit -n`, `RLIMIT_NOFILE`)에 묶인다. 수백 개 패널을 열면 한계에 닿을 수 있어, 패널 수에 합리적 상한·경고를 둔다.
- **프로세스/메모리 한계**: 사용자 `RLIMIT_NPROC`, cgroup(user.slice) 메모리/CPU 제한 하에서 동작. 데몬이 스크롤백을 메모리에 들고 있으므로 패널 수×스크롤백 라인에 메모리가 비례 → 상한 설정.
- **소켓/권한**: IPC 소켓은 `$XDG_RUNTIME_DIR/tuimux/`(없으면 `~/.tuimux/`, 권한 `0700`) 아래 `0600`. 타 사용자 attach 차단(NFR-5).
- **고아 데몬 누수**: 모든 세션이 비어도 데몬이 안 죽으면 누수. → "마지막 세션 종료 후 N초 유휴면 데몬 자동 종료" 옵션, 그리고 `tuimux kill-server`.
- **`$XDG_RUNTIME_DIR` 소멸**: 이 디렉터리는 로그아웃 시 사라질 수 있다(소켓도 함께). linger가 켜져 있으면 유지된다. 미설정 시 `~/.tuimux` 폴백으로 회피.

### 9.6 트레이드오프 요약 (지속성)
| 방식 | 사용자 권한만으로? | "연결 끊김" 생존 | "완전 로그아웃" 생존 | 설정 비용 | 비고 |
|---|---|---|---|---|---|
| 9.3-(1) 단순 데몬화 (`setsid`) | ✅ | ✅ | `KillUserProcesses=no`면 ✅ / `yes`면 ❌ | 없음 | tmux/screen 방식, MVP 기본 |
| 9.3-(2) user systemd service | ✅ | ✅ | linger 없으면 정책 종속 | 낮음 | 운영성↑, 재시작/journal |
| 9.3-(3) service + linger | ✅ | ✅ | ✅ (정책 무관) | 중간(`enable-linger`) | 가장 견고, 권장 |
| 옵션 2: tmux 백엔드 | ✅ | ✅ | tmux 데몬과 동일(통상 ✅) | tmux 설치 필요 | 지속성은 tmux가 보장 |

**결론**: tuimux는 기본적으로 **9.3-(1) 단순 데몬화**로 동작해 "끊겨도 생존"을 무설정으로 제공하고, "로그아웃 후에도 무조건 생존"을 원하는 사용자에게 **user systemd service + linger**(`loginctl enable-linger`)를 1급 지원/문서화한다. root 권한은 필요 없다.

## 10. 기술 의사결정 (Technical Decisions)

- **TD-1 언어/런타임**: Rust. 메모리 안전 + 단일 바이너리 + 우수한 TUI 생태계.
- **TD-2 프로세스 모델**: **서버(데몬)/클라이언트 분리** 확정. 지속성과 다중 attach의 전제.
- **TD-3 IPC**: 유닉스 도메인 소켓 + 길이-프리픽스 프레이밍 메시지. 프로토콜은 버전 필드 포함(전·후방 호환 여지). 직렬화는 경량 바이너리(후보: bincode/postcard, 미확정).
- **TD-4 TUI 스택(후보)**: 렌더 `ratatui`, 백엔드/입력 `crossterm`(마우스·키·리사이즈 이벤트). 셀 그리드/VT는 §8 옵션에 따라 `vte`/`vt100`/`wezterm-term` 중 평가 후 채택.
- **TD-5 입력 정책**: prefix 없음. 전역=`Alt`(Meta) 조합 + 마우스 + 명령 팔레트. 패널 내부 앱과의 키 충돌은 "전역은 Alt, 패널 통과는 그 외"로 분리.
- **TD-6 마우스**: SGR(1006) 확장 마우스 모드로 큰 좌표·드래그·휠 지원. 패널 내부 앱이 마우스를 요구하면(예: vim, htop) 해당 패널에선 마우스 이벤트를 앱으로 패스스루하고, tuimux UI(탭/경계)는 별도 영역에서 가로챈다.
- **TD-7 지속성 기본값**: 단순 데몬화(setsid). systemd user service + linger를 문서/패키징에서 권장 경로로 제공(§9).
- **TD-8 스크롤백**: 메모리 내 링버퍼, 패널당 기본 라인 수 상한(설정 가능). 디스크 영속화는 후순위(NG6).
- **TD-9 설정**: 소수 옵션만, 단일 TOML(`~/.config/tuimux/config.toml`). "기본값으로 충분" 원칙.
- **TD-10 타깃 OS**: Linux 우선(개발/검증), macOS best-effort(개념적으로 동일, logind 대신 launchd 차이 존재), Windows 후순위(ConPTY).
- **TD-11 아키텍처 경로**: control-mode 스파이크로 UX 검증 → 네이티브 엔진으로 본구현(§8 권고).

## 11. 리스크 (Risks)

| ID | 리스크 | 영향 | 완화책 |
|---|---|---|---|
| RK-1 | 자체 VT 에뮬레이션의 호환성 버그(앱 깨짐) | 높음 | 성숙한 크레이트 채택, xterm 테스트 코퍼스/vttest 회귀, 옵션 2 스파이크로 기준 비교 |
| RK-2 | `KillUserProcesses=yes` 환경에서 로그아웃 후 세션 소멸 | 높음 | linger/ user service 1급 지원 + 명확한 문서·진단 명령(`tuimux doctor`) |
| RK-3 | No-prefix `Alt` 키가 일부 터미널/OS에서 안 먹힘 | 중간 | 명령 팔레트·마우스 메뉴라는 대체 경로 항상 제공, 리매핑 허용 |
| RK-4 | 마우스 패스스루 충돌(내부 앱 vs tuimux UI) | 중간 | 영역 기반 분리(테두리/탭은 tuimux, 콘텐츠는 앱), 토글로 강제 패스스루 |
| RK-5 | 다중 클라이언트 리사이즈 충돌(가장 작은 화면에 맞춤?) | 중간 | MVP는 단순 정책(최신 attach 기준 또는 최소 공통), 이후 per-pane 정책 |
| RK-6 | 옵션 2 선택 시 tmux 버전/`-CC` 차이 | 중간 | 지원 버전 명시, control-mode를 격리 레이어로 캡슐화 |
| RK-7 | 데몬 누수/좀비, FD·PTY 고갈 | 중간 | 유휴 자동 종료 옵션, 한계 경고, `kill-server` 제공 |
| RK-8 | 단일 바이너리 목표 vs 옵션 2의 tmux 의존 모순 | 중간 | 장기 경로를 네이티브로 확정해 의존 제거 |
| RK-9 | "단순함" 약속 vs 기능 요구 압력(스코프 크리프) | 중간 | Non-Goals를 강하게 유지, 옵션은 소수·기본값 우선 |

## 12. 마일스톤 (Milestones)

- **M0 — 스파이크/검증 (Weeks 1–2)**
  - tmux control-mode(옵션 2a) 위에 "우측 탭 + 마우스 클릭 전환 + detach 생존" 최소 프로토타입.
  - 목적: UX 가설(prefix-free·우측 탭·마우스)과 지속성 시나리오(§7 DoD)를 빠르게 입증.
- **M1 — 네이티브 코어 (Weeks 3–6)**
  - 서버/클라이언트 분리, 유닉스 소켓 IPC, PTY 1개 + VT 파서 통합으로 단일 패널 셸 렌더.
  - `setsid` 데몬화 + detach/attach + SSH 끊김 생존(§9.2/9.3-1) 동작.
- **M2 — 멀티플렉싱/레이아웃 (Weeks 7–9)**
  - 세션/창/패널 3계층, 분할·닫기·리사이즈, 우측 세로 탭, 상태 바.
- **M3 — 마우스/팔레트/도움말 (Weeks 10–11)**
  - 전체 마우스(클릭/드래그/휠/우클릭 메뉴), 명령 팔레트, 도움말 오버레이, 스크롤백.
- **M4 — 프로세스 가시성 (Weeks 12–13)**
  - PROCS 패널: 실행 중 표시 + 종료 상태 + 신호 전송(R-X1~3), `/proc` 연동.
- **M5 — 지속성 강화/패키징 (Weeks 14–15)**
  - systemd user service 유닛 + linger 문서/`tuimux doctor` 진단, 권한/소켓 하드닝, 한계·자동종료.
- **M6 — MVP 안정화/릴리스 (Week 16)**
  - 회귀 테스트(vttest 류), 성능(지연/유휴 CPU) 점검, 문서, v0.1 단일 바이너리 릴리스.

(주: 기간은 추정치이며 팀 규모/검증 결과에 따라 조정. M0 결과가 옵션 1 전면 채택 여부를 확정한다.)
