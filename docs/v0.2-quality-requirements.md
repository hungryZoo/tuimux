# tuimux v0.2-quality terminal wrapper 요구사항 / QA 인수 기준

> 참고: 2026-06-12 현재 `capture-pane` 기반 main pane은 PTY + `vt100` 기반 terminal surface로 교체되었습니다. 최신 구현 기준은 [SRS v0.9](./srs.md)와 [SDD v0.9](./sdd.md)를 우선합니다.

- **문서 버전**: 0.1 Draft
- **작성일**: 2026-06-09
- **대상 릴리스**: v0.2 quality milestone
- **범위**: 제품/QA 관점 spec review. 구현 설계 또는 코드 변경은 포함하지 않는다.
- **핵심 결론**: tuimux는 `tmux`를 단순 실행하는 launcher가 아니라, `tmux` server/backend 위에 tuimux chrome/sidebar/main terminal surface를 제공하는 **TUI wrapper 제품**이어야 한다.

---

## 1. 제품 방향 재정의

### 1.1 문제 배경

v0.1.9는 `tuimux` 기본 실행 시 plain tmux client가 아니라 tuimux ratatui TUI가 뜨는 상태를 유지하면서, main pane에 ANSI 색상 파싱·blank row 보존·viewport resize 동기화를 추가했다. 그러나 장기적으로는 `capture-pane` snapshot + `send-keys` input bridge를 control-mode/terminal surface 구조로 발전시켜야 한다. 이 구조는 기본 UI 복구에는 충분하지만, 사용자가 기대하는 “터미널 wrapper” 품질에는 부족하다.

현재 보고된 핵심 품질 문제:

1. `clear` 이후 이전 글자가 main pane에 잔상처럼 남는다.
2. shell ANSI 색상/스타일이 tuimux main pane에 제대로 반영되지 않는다.
3. resize 시 main area가 tmux pane 크기와 정밀하게 동기화되지 않아 표시/줄바꿈/빈 공간이 어색하다.
4. main pane이 full terminal surface라기보다 주기적으로 긁어온 text preview처럼 보인다.

### 1.2 v0.2 제품 원칙

- **TUI wrapper first**: `tuimux` 기본 실행은 항상 tuimux UI여야 한다. plain `tmux attach`는 opt-in fallback일 뿐이다.
- **tmux fidelity without UI removal**: tmux-native 품질 문제를 해결하기 위해 tuimux UI를 없애면 안 된다.
- **Main pane is a terminal surface**: v0.2부터 main pane은 단순 text snapshot이 아니라 shell 화면 상태, 색상, clear, resize를 신뢰할 수 있게 반영해야 한다.
- **Control-mode direction**: 장기적으로 iTerm2처럼 `tmux -CC`/control-mode 또는 동등한 protocol-level integration을 기준 방향으로 둔다.
- **Honest milestone**: v0.2에서 full iTerm2 수준을 한 번에 완성하지 못하더라도, 현재 사용자 불만 3종(clear 잔상, ANSI color, resize)을 인수 기준으로 고정한다.

## 2. 릴리스 목표 / 비목표

### 2.1 목표

- **G1. TUI 정체성 유지**: `tuimux` 실행 시 sidebar/session/window UI가 유지된다.
- **G2. clear 무잔상**: shell `clear`, `tput clear`, alternate screen 진입/해제 후 main pane에 이전 text가 남지 않는다.
- **G3. ANSI 색상 fidelity**: 최소 16색, 256색, truecolor foreground/background, bold/underline/reverse를 화면에 반영한다.
- **G4. resize fidelity**: 터미널 크기 변경 시 tuimux layout과 tmux pane size가 동기화되고, wrapping/빈 영역/스크롤 위치가 깨지지 않는다.
- **G5. 입력 bridge 품질**: 일반 문자, Enter, Backspace, Tab, arrows, Ctrl 조합, paste가 중복/누락 없이 pane으로 전달된다.
- **G6. 회귀 테스트 가능성**: 위 항목은 사람이 보는 smoke test뿐 아니라 가능한 범위에서 자동화 가능한 PTY/recording 기반 acceptance test로 고정한다.

### 2.2 비목표

- v0.2에서 tmux 자체를 재구현하지 않는다.
- v0.2에서 모든 terminal escape sequence를 완벽히 지원한다고 주장하지 않는다.
- v0.2에서 tuimux UI를 제거하고 native tmux client로 대체하지 않는다.
- v0.2에서 multi-pane split 편집기 전체 UX를 완성하지 않는다. 단, 현재 active pane의 terminal surface 품질은 릴리스 차단 기준으로 본다.

## 3. 우선순위

### P0: 릴리스 차단

1. **Default TUI 회귀 금지**
   - `tuimux`는 tuimux TUI를 띄워야 한다.
   - plain tmux attach는 `--native-client` 등 명시 옵션에서만 가능해야 한다.

2. **clear 잔상 제거**
   - `clear` 실행 후 이전 prompt/output이 main pane visible area에 남으면 v0.2 release blocker다.

3. **ANSI 색상 기본 반영**
   - shell prompt, `ls --color`, `printf` SGR 색상이 monochrome text로만 보이면 release blocker다.

4. **resize 동기화**
   - host terminal resize 후 main pane width/height와 tmux pane geometry가 어긋나거나, 이전 frame 조각이 남으면 release blocker다.

5. **상태 복구**
   - panic, Ctrl-C, q, detach 후 raw mode/alternate screen/mouse capture가 복구되어야 한다.

### P1: v0.2 강력 권장

- 256색/truecolor sample rendering.
- basic cursor position/cursor visibility 표시.
- large output tailing 성능 안정성.
- mouse click focus 및 sidebar hover가 main pane rendering과 충돌하지 않음.
- paste bracketed mode 또는 안전한 paste handling.

### P2: 후속 milestone

- tmux control-mode full client.
- multiple panes/splits의 pixel/region 정확한 rendering.
- copy-mode/search/scrollback UI.
- tmux statusline와 tuimux chrome의 통합 설정.

## 4. 기능 요구사항

### 4.1 실행 모드

- **FR-RUN-1 [P0]** 인자 없는 `tuimux`는 tuimux TUI를 실행한다.
- **FR-RUN-2 [P0]** `--native-client`는 fallback이며 help/docs에서 기본 경로로 소개하지 않는다.
- **FR-RUN-3 [P0]** fallback 사용 여부는 로그/테스트에서 식별 가능해야 한다.
- **FR-RUN-4 [P0]** tmux server/session/window 조작은 실제 tmux backend에 반영되어야 한다.

### 4.2 main pane rendering

- **FR-TERM-1 [P0]** main pane은 active tmux pane의 visible screen state를 표현해야 한다.
- **FR-TERM-2 [P0]** `clear`, `reset`, `tput clear`, `printf '\033[2J\033[H'` 이후 지워진 cell이 이전 content를 표시하면 안 된다.
- **FR-TERM-3 [P0]** line shorter than previous line인 경우 오른쪽 잔여 cell이 이전 line 글자를 유지하면 안 된다.
- **FR-TERM-4 [P0]** SGR color/style 정보를 text와 함께 보존/렌더해야 한다. 최소 16색, bold, underline, reverse는 P0다.
- **FR-TERM-5 [P1]** 256색과 truecolor foreground/background를 가능한 터미널 capability 내에서 반영한다.
- **FR-TERM-6 [P0]** main pane 내부 content는 tuimux border/sidebar/modal 렌더링 영역을 침범하지 않는다.
- **FR-TERM-7 [P0]** shell cursor 위치는 최소한 입력 위치가 사용자가 이해 가능하게 표시되어야 한다.
- **FR-TERM-8 [P1]** alternate screen app(`vim`, `less`, `top`) 진입/해제 시 visible surface가 크게 깨지지 않는다.

### 4.3 tmux control-mode / protocol 방향

- **FR-CM-1 [P1]** v0.2 구현 선택지는 `tmux capture-pane`만 반복 호출하는 구조의 한계를 명시하고, control-mode renderer로 이행 가능한 boundary를 둔다.
- **FR-CM-2 [P1]** control-mode 사용 시 `%output`, `%layout-change`, `%window-add`, `%window-close`, `%session-changed`, `%pane-mode-changed`류 event를 제품 state로 반영해야 한다.
- **FR-CM-3 [P1]** protocol parser/renderer는 tuimux chrome과 main terminal surface를 분리해 테스트 가능해야 한다.
- **FR-CM-4 [P2]** iTerm2 `tmux -CC` 수준의 장기 목표는 “tmux server의 screen model을 custom UI에 정확히 투영”하는 것이다. 단순 native client embed가 아니다.

### 4.4 resize

- **FR-RESIZE-1 [P0]** host terminal resize event 수신 시 ratatui layout을 즉시 재계산한다.
- **FR-RESIZE-2 [P0]** main pane inner rect 크기를 tmux active pane/window size에 반영한다.
- **FR-RESIZE-3 [P0]** resize 후 첫 repaint에서 이전 frame 잔상 없이 전체 visible area를 다시 그린다.
- **FR-RESIZE-4 [P0]** 가로 shrink/expand 시 shell line wrapping이 tmux와 tuimux 표시 간에 일관되어야 한다.
- **FR-RESIZE-5 [P1]** 빠른 연속 resize에도 flicker/lag가 사용 불가능 수준으로 증가하지 않는다.

### 4.5 input

- **FR-IN-1 [P0]** terminal input mode에서 문자 입력은 한 번만 전달된다. key repeat/release 중복은 없어야 한다.
- **FR-IN-2 [P0]** Enter, Backspace, Delete, Tab, Escape, arrows, Home/End가 shell에서 기대대로 동작한다.
- **FR-IN-3 [P0]** Ctrl-C/Ctrl-D/Ctrl-L 등 shell control key가 main pane에 전달될 때 tuimux global shortcut과 충돌하지 않아야 한다.
- **FR-IN-4 [P0]** F12 또는 명시된 escape key는 terminal input mode에서 navigation mode로 안정적으로 복귀한다.
- **FR-IN-5 [P1]** paste는 줄 누락/중복/순서 뒤바뀜 없이 전달된다.

### 4.6 UI chrome / sidebar

- **FR-UI-1 [P0]** right sidebar(Session, Detach, WINDOWS, `+ new`)는 main terminal 품질 개선 후에도 유지된다.
- **FR-UI-2 [P0]** modal/session dialog는 main pane repaint와 겹쳐 잔상을 만들지 않는다.
- **FR-UI-3 [P0]** mouse hover/click target은 resize 후에도 실제 좌표와 일치해야 한다.
- **FR-UI-4 [P1]** main pane rendering이 바쁘더라도 sidebar 조작 latency는 사용자가 체감 가능한 수준으로 유지되어야 한다.

## 5. 비기능 요구사항

- **NFR-PERF-1 [P0]** idle 상태 CPU 사용량은 과도한 polling으로 치솟지 않아야 한다.
- **NFR-PERF-2 [P1]** 초당 수천 줄 출력(`yes`, build log 등)에서도 UI가 완전히 멈추지 않는다.
- **NFR-SAFE-1 [P0]** crash/exit 후 terminal raw mode, cursor, alternate screen, mouse capture를 복구한다.
- **NFR-COMPAT-1 [P0]** macOS iTerm2/Terminal.app, Linux xterm-compatible terminal에서 기본 동작을 검증한다.
- **NFR-COMPAT-2 [P0]** tmux minimum supported version을 명확히 고정하고 doctor가 경고한다.
- **NFR-OBS-1 [P1]** rendering backend mode(snapshot/control-mode/fallback), tmux version, TERM/COLORTERM 정보를 debug log 또는 doctor에서 확인할 수 있다.

## 6. Acceptance tests

아래 테스트는 “통과해야 릴리스 가능”한 v0.2-quality 기준이다. 자동화 난이도가 높은 항목은 PTY recording + golden snapshot 또는 수동 QA checklist로 시작하되, 반복 회귀가 난 항목(clear/color/resize)은 자동화를 우선한다.

### 6.1 기본 TUI/회귀

- **AC-RUN-1 [P0]** `cargo test --quiet` 통과.
- **AC-RUN-2 [P0]** `tuimux` PTY 실행 시 화면에 tuimux UI 식별 문자열(`Session`, `Detach`, `WINDOWS`, `New Session`)이 나타난다.
- **AC-RUN-3 [P0]** `tuimux` PTY 실행 시 plain tmux attach 화면만 보이면 실패다.
- **AC-RUN-4 [P0]** `tuimux --native-client`에서만 native tmux client path를 탄다.

### 6.2 clear 잔상

- **AC-CLEAR-1 [P0]** clean tmux session에서 `printf 'AAAAA\nBBBBB\n'; clear` 실행 후 visible main pane에 `AAAAA`/`BBBBB`가 남지 않는다.
- **AC-CLEAR-2 [P0]** `printf '1234567890\rabc\n'`처럼 짧은 문자열이 긴 문자열을 덮는 경우 line tail `4567890`이 잔상으로 남지 않는다.
- **AC-CLEAR-3 [P0]** `printf '\033[2J\033[Hafter-clear'` 후 main pane에는 `after-clear`와 정상 prompt만 보이고 이전 content가 없다.
- **AC-CLEAR-4 [P0]** session modal을 열었다 닫아도 modal border/text가 main pane에 남지 않는다.

### 6.3 ANSI color/style

- **AC-COLOR-1 [P0]** `printf '\033[31mRED\033[0m \033[32mGREEN\033[0m \033[34mBLUE\033[0m\n'` 출력 시 각 text가 서로 다른 foreground color로 렌더된다.
- **AC-COLOR-2 [P0]** `printf '\033[1mBOLD\033[0m \033[4mUNDER\033[0m \033[7mREV\033[0m\n'` 출력 시 bold/underline/reverse가 반영되거나 terminal capability 한계가 명시적으로 degrade된다.
- **AC-COLOR-3 [P1]** `printf '\033[38;5;202mORANGE256\033[0m\n'`가 256색에 가깝게 렌더된다.
- **AC-COLOR-4 [P1]** `printf '\033[38;2;120;80;200mTRUECOLOR\033[0m\n'`가 truecolor 또는 가장 가까운 fallback color로 렌더된다.
- **AC-COLOR-5 [P0]** shell prompt color가 command output color와 섞이거나 reset 누락으로 sidebar 색을 오염시키지 않는다.

### 6.4 resize

- **AC-RESIZE-1 [P0]** PTY를 120x40에서 80x24로 줄인 후 main pane border/sidebar가 깨지지 않고, tmux pane width가 main pane inner width와 일치한다.
- **AC-RESIZE-2 [P0]** 80x24에서 140x50으로 늘린 후 이전 오른쪽/아래쪽 frame 잔상이 없다.
- **AC-RESIZE-3 [P0]** 긴 줄 출력 후 가로 resize를 반복해도 wrapping 결과가 tmux native client와 의미상 일치한다.
- **AC-RESIZE-4 [P0]** resize 직후 mouse click target(Session/Detach/window row/close/new)이 화면 좌표와 일치한다.
- **AC-RESIZE-5 [P1]** 10회 이상 연속 resize 중 panic, terminal corruption, stuck raw mode가 없다.

### 6.5 input

- **AC-IN-1 [P0]** terminal input mode에서 `echo hello` 입력 후 tmux pane에 정확히 한 번 `hello`가 실행된다.
- **AC-IN-2 [P0]** Backspace로 `hellp`를 `hello`로 고친 뒤 Enter 시 shell output이 `hello`다.
- **AC-IN-3 [P0]** Ctrl-L은 shell clear로 동작하며 AC-CLEAR 기준을 만족한다.
- **AC-IN-4 [P0]** arrow up/down으로 shell history 탐색이 가능하다.
- **AC-IN-5 [P0]** F12 후 `q`는 shell에 입력되지 않고 tuimux navigation quit로 처리된다.
- **AC-IN-6 [P1]** multi-line paste가 줄 누락/중복 없이 실행된다.

### 6.6 fullscreen/alternate-screen apps

- **AC-FULL-1 [P1]** `less` 실행 후 화면, scroll, quit가 main pane에 정상 반영된다.
- **AC-FULL-2 [P1]** `vim` 또는 `nvim` 실행/종료 시 editor 화면과 복귀 shell 화면이 심각하게 깨지지 않는다.
- **AC-FULL-3 [P1]** `top`/`htop`류 반복 업데이트 화면에서 clear 잔상과 색상 reset 문제가 없다.

### 6.7 robustness

- **AC-ROBUST-1 [P0]** tuimux 종료 후 host terminal에서 입력 echo, cursor, colors가 정상이다.
- **AC-ROBUST-2 [P0]** tmux server가 중간에 kill되어도 tuimux가 panic하지 않고 recoverable error/status를 표시한다.
- **AC-ROBUST-3 [P1]** 빠른 output 중 sidebar click/window switch가 deadlock 없이 처리된다.

## 7. 권장 QA 시나리오

### Scenario A: 사용자가 처음 실행

1. tmux server 없는 상태에서 `tuimux` 실행.
2. tuimux session 자동 생성 확인.
3. Session modal, Detach, WINDOWS, `+ new` 확인.
4. main pane 클릭 후 `echo hello` 실행.
5. F12로 navigation 복귀, q로 종료.

합격 기준: UI가 plain tmux가 아니며 shell command가 실제 tmux pane에서 실행되고 종료 후 terminal 상태가 정상이다.

### Scenario B: clear 잔상 재현/검증

1. main pane에서 여러 줄/긴 줄 출력.
2. `clear`, Ctrl-L, `tput clear`, raw CSI clear sequence 실행.
3. modal open/close, window switch 후 다시 확인.

합격 기준: 이전 글자/테두리/색상 잔상이 visible area에 없다.

### Scenario C: color fidelity

1. 16색 sample 출력.
2. 256색 sample 출력.
3. truecolor gradient 또는 RGB sample 출력.
4. `ls --color=always`, colored prompt 확인.

합격 기준: 최소 16색과 스타일은 구분되고, 색상 reset이 UI chrome을 오염시키지 않는다.

### Scenario D: resize stress

1. 120x40 → 80x24 → 140x50 → 60x20 → 100x30 반복.
2. 각 단계에서 long line, prompt, sidebar, modal, mouse target 확인.
3. resize 직후 shell command 입력.

합격 기준: layout과 tmux pane geometry가 동기화되고 잔상/좌표 오차가 없다.

### Scenario E: native fallback 경계

1. `tuimux` 기본 실행 확인.
2. `tuimux --native-client` 실행 확인.
3. docs/help에서 fallback이 기본 UX로 오해되지 않는지 확인.

합격 기준: tuimux 제품 UI가 기본이며 fallback은 명시 opt-in이다.

## 8. 현재 코드 기준 위험 메모

현재 v0.1.9 코드/문서 기준으로 확인되는 위험은 다음과 같다.

- `tmux capture_pane(session, width, height)`는 현재 width/height를 실질적으로 사용하지 않는 구조다. resize fidelity 요구와 직접 충돌할 가능성이 높다.
- `capture-pane -p` text parsing은 ANSI style/screen cell 정보를 충분히 보존하지 못할 수 있다. color acceptance를 만족하려면 capture option 변경만으로 충분한지 검증이 필요하다.
- `parse_capture`는 trailing empty line을 제거한다. 실제 terminal screen의 blank cells/cleared lines와 “없는 line”을 구분하지 못하면 clear 잔상/resize blank area 문제가 반복될 수 있다.
- ratatui `Paragraph`에 raw string line을 넣는 현재 형태는 per-cell style/color fidelity를 표현하기 어렵다.
- 120ms polling snapshot 방식은 high-output/alternate-screen app에서 flicker 또는 stale frame을 만들 수 있다.

이 메모는 구현 지시가 아니라 QA 관점의 risk register다.

## 9. 릴리스 판정 기준

v0.2-quality 릴리스는 다음을 모두 만족해야 한다.

1. P0 acceptance test 전부 통과.
2. clear/color/resize 관련 known blocker가 0개.
3. 기본 실행이 tuimux TUI임을 PTY smoke test로 확인.

## 10. 최종 PM 판단

v0.2의 성공은 “tmux를 실행할 수 있다”가 아니라 “tmux backend를 사용하면서 tuimux UI 안에서 terminal처럼 믿고 쓸 수 있다”로 판단해야 한다. 따라서 default TUI 유지, clear 무잔상, ANSI 색상 반영, resize 정합성은 선택 기능이 아니라 제품 정체성을 지키는 릴리스 차단 요구사항이다.
